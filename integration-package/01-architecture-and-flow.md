# Архитектура, флоу и ограничения (Stage 1)

## 1) Роль сервиса в продукте

Сервис стоит перед основным продуктом как «голосовой вход». Его задача:

1. Принять аудио из браузера.
2. Распознать русскую речь.
3. Вернуть JSON вида `{ "text": "..." }`.

Он **не** делает кухонную логику, не генерирует дизайн и не хранит бизнес-данные основного сервиса.

## 2) Что мы сознательно выбрали

1. Один бизнес-эндпойнт: `POST /v1/transcribe`.
2. Синхронный режим (без стриминга/WebSocket/gRPC).
3. Push-to-talk сценарий: пользователь зажал кнопку -> отпустил -> фронт отправил один файл.
4. Жесткий серверный лимит длительности: **29 секунд**.
5. CORS в stage 1: `*`.
6. Простая авторизация: `X-API-Key`.
7. Провайдер STT инкапсулирован за интерфейсом (`SttProvider`) для будущей замены.

## 3) Рабочий флоу (Happy Path)

1. Браузер пишет короткий клип (обычно WebM/Opus, возможен MP3/WAV и т.д.).
2. Основной сервис/клиент отправляет `multipart/form-data` на `POST /v1/transcribe`.
3. STT-сервис проверяет:
   - API key,
   - формат запроса,
   - наличие и размер файла,
   - длительность (<= 29с),
   - корректность аудио.
4. При необходимости сервис конвертирует аудио в формат, подходящий под STT-провайдера.
5. Сервис получает финальный текст от STT-провайдера.
6. Возвращает в ответе только `{ "text": "..." }`.
7. Основной сервис продолжает ваш текущий text-first пайплайн без изменений.

## 4) Ограничения, с которыми нужно работать

1. Лимит длительности: >29с -> `422 audio_too_long`.
2. Лимит размера payload: ограничен серверной настройкой (`MAX_UPLOAD_BYTES`, по умолчанию 1 MB).
3. Поддерживаются распространённые browser MIME-типы:
   - `audio/webm`, `audio/ogg`, `audio/wav`, `audio/mpeg`, `audio/mp4`, `audio/x-m4a`, `audio/aac`.
4. При некорректном/битом контенте возвращается JSON-ошибка (`corrupted_audio`, `malformed_request` и т.д.).
5. При проблемах внешнего провайдера возвращаются `upstream_error` / `upstream_timeout`.
6. При всплеске запросов возможны `rate_limited` (`429`) и `service_busy` (`503`).

## 5) Управление rollout (ограниченный запуск)

Чтобы быстро отключать «проблемную точку» без остановки сервиса:

1. Используются ключи по локациям (`SERVICE_API_KEYS`: `clientId:key`).
2. Есть админ-эндпойнты для revoke/restore конкретного `clientId`.
3. Есть метрики по клиентам (`/admin/metrics`) для раннего выявления аномального использования.

## 6) Что ожидается от разработчика основного сервиса

1. Делать вызов `POST /v1/transcribe` после завершения записи.
2. Обрабатывать не только `200`, но и ожидаемые JSON-ошибки.
3. При частых ошибках/нагрузке иметь fallback: отключить голос и использовать текстовый ввод.
4. Хранить и передавать ключи безопасно (не логировать, не публиковать).

## 7) Что не входит в stage 1

1. Потоковое распознавание.
2. Диаризация/сложная пост-обработка текста.
3. TTS.
4. Бизнес-логика дизайна кухни.
